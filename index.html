How It Works

How Subagents Fit Together

Instead of one agent reading, researching, building, and reviewing in sequence inside a single context window, the main agent delegates each concern to a specialist and collects the results.

Explorer, Researcher, and Builder run concurrently — none of them depend on each other. Reviewer runs after, because it needs Builder's diff to check. Each specialist keeps its own context window — the noisy intermediate work (grep output, doc fetches, file diffs) never pollutes the main conversation.

Cowork subagents are isolated, disposable Claude instances: each gets a clean context window, does its piece, and reports its result back to you. They can't message each other or coordinate mid-task — you're always the one relaying between them.

Each worker still reports back to the main agent automatically, same as Claude Code. The dashed break marks the one real difference: workers never see each other's instructions or talk mid-task.

How we know: this was verified by reading Cowork's own session log files on disk — not by asking it to describe its own capabilities. The logs showed workers running concurrently on fully isolated instructions, and a direct test confirmed agent-to-agent coordination isn't supported here.

Mind what you hand off. A subagent's prompt and any files it reads become part of its own context — don't route customer PII, credentials, or anything sensitive into scaffolding you haven't reviewed. Scope tool access (read-only agents like explorer/reviewer) rather than defaulting every agent to full file access.

Cowork doesn't have named specialist roles the way Claude Code does — instead, match the shape of the work to how you split it.

example-prompt.txt 
I have 40 win-loss interview transcripts in this folder. Split them into batches of 5, spawn one subagent per batch, and have each subagent write a one-paragraph win/loss theme summary for its batch. Save each subagent's output to a distinct file named batch-1-summary.md, batch-2-summary.md, etc. so results don't overwrite each other. When all batches are done, compile the summaries into one combined doc.
Three more things to watch for:
  • If a split works well for something you do repeatedly (interview research, Gong call batches), turn it into a Skill instead of re-explaining the batch size every time — the skill-creator skill can help.
  • If multiple subagents will each produce a file, give each one an explicit, distinct filename up front. Left to just "save your result," they can silently overwrite each other.
  • Each subagent is a full request, not a free clone — a 40-way split costs roughly 40x a single request's tokens. Batch coarser (fewer, larger subagents) if you're watching usage, not finer.
  • Subagents pull real customer or account data into their own context the moment you point them at it — don't hand off anything sensitive (PII, credentials, deal terms) to a subagent whose output you haven't scoped or reviewed.

Parallel wall-clock ≈ the slowest concurrent task (build, 8m) + dispatch overhead + the review step that has to wait for the build to finish. Explorer and researcher's time is "free" here because it overlaps with the longest task. Real savings depend entirely on how independent your subtasks are — if everything shares state or one step blocks the next, parallelizing buys you nothing. In this scenario, running the independent steps in parallel cuts wall-clock time by roughly 48% — that's the shape of the benefit, not a guarantee; measure your own tasks before treating it as a rule.

When To Use

When To Actually Use This

The common failure mode isn't spawning the wrong agent — it's not spawning at all and grinding through everything in one context.

Spawn A Subagent When

  • You're about to open a 3rd unfamiliar file just to answer one question
  • You're running the same grep/glob a 3rd time to map something
  • A doc fetch or web search would dump a lot of noise into your context
  • 2+ files need building and don't share state — run builders in parallel
  • You just finished a multi-file build and are about to call it done

Skip Spawning When

  • It's a simple edit to 1–3 files you already know
  • It's a conversation, analysis, or planning turn with no file ops
  • You'd need to pause and ask a question mid-task — subagents can't do that
  • The whole task is ~5 minutes — orchestration overhead isn't worth it
Escape hatch: none of this is a mandate. If delegating is making a task slower, more confusing, or producing worse output than just doing it yourself, stop and do it directly.

The common failure mode is the same: not splitting at all, or splitting too coarsely. Just as important is knowing what Cowork subagents flatly can't do.

Split The Work When You Notice

  • You're about to look up or summarize the 4th+ similar record in a row, one at a time
  • A request contains "each," "every," or "all N" and N is more than a handful
  • You're drafting the same type of output (summary, outreach email, brief) repeatedly in sequence

What This Can't Do

  • Agent-to-agent messaging — subagents report only to you, never to each other
  • Real-time coordination mid-task — no live back-and-forth between workers
  • There's no way to work around this — treat it as a hard limit, not something to route around with cleverer prompting
Escape hatch: none of this is a mandate. If splitting the work is making things slower or more confusing than just doing it directly, stop and do it directly.
